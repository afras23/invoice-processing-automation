# ADR 001: Multi-Format Document Parsing Strategy

**Date:** 2026-03-01
**Status:** Accepted

## Context

Invoice documents arrive in multiple formats from different clients and systems:
- PDFs generated by accounting software (structured, extractable text)
- Scanned PDFs and image files (require OCR or vision models)
- CSV exports from ERP systems
- Plain text pastes from email bodies

The parsing layer must produce a uniform `ParsedDocument` with extracted text and a stable content hash, regardless of the source format.

## Decision

Implement a separate parser class per format, dispatched by a format-detection function that inspects file extension and MIME type. All parsers return the same `ParsedDocument` Pydantic model.

Format dispatch:
- `.pdf` → `PDFParser` using `pdfplumber`
- `.csv`, `.tsv` → `CSVParser` with auto-delimiter detection
- `.png`, `.jpg`, `.jpeg`, `.tiff` → `ImageParser` (returns extracted text via vision prompt or OCR)
- everything else → `TextParser` (treat as UTF-8 plain text)

The content hash is computed over **raw file bytes** before parsing, not over extracted text. This ensures:
- The hash is deterministic regardless of parser version changes
- Two files with identical bytes always produce the same hash
- Text extraction failures do not affect deduplication

## Alternatives Considered

### Single parser with format branches
Simpler in terms of file count, but violates single responsibility. Adding a new format (e.g., XLSX) would require editing the same class as PDF logic.

### Hash over extracted text
More "semantic" — two invoices with the same content but different whitespace would deduplicate. Rejected because it introduces parser dependency into the deduplication layer, and because we want deterministic hashes regardless of parser changes.

### Apache Tika or similar polyglot parser
Avoids per-format code, but adds a JVM dependency. `pdfplumber` plus standard library CSV handling is sufficient for the formats in scope and has no external service dependency.

## Consequences

**Positive:**
- New formats can be added by implementing a new parser class and registering it in the dispatch table — no existing code changes
- Hash over bytes means deduplication is independent of parsing quality
- Each parser can be unit-tested with sample files without touching the rest of the pipeline

**Negative:**
- Image invoices currently use a basic path; full OCR accuracy requires either a vision model call (adds latency and cost) or a local OCR library (adds a heavyweight dependency)
- CSV auto-delimiter detection can misfire on edge-case files; the fallback is to treat the file as plain text
